## Refactor-Agent Vokabular-Erweiterung — wenn zugrundliegender Workflow repariert ist

Bisheriger Versuch (RQ-19-refactor-vocab-extended) invalide. Wenn Re-Setup ansteht:
ein neuer Block in `.claude/agents/refactor.md` zwischen "Naming Evaluation"
und "Absolute Priority Premise (APP)" — wortlaut-fertig:

```markdown
### Complexity Awareness (Second Refactoring Priority)

After naming, look at the function's structural complexity through two complementary lenses:

- **Cyclomatic Complexity (McCabe)** — how many independent paths run through this function? Each `if`, `else if`, `case`, `&&`, `||`, `?:`, `for`, `while`, and `catch` adds a path. A function with many branches has many paths to test and many ways to be wrong.
- **Cognitive Complexity (SonarSource)** — how hard is this function to *understand*? Nesting is weighted heavier than flat control flow. A `for` inside an `if` inside a `while` is cognitively heavier than three sequential flat blocks doing the same work, even when McCabe is equal.

When a function feels dense, ask:
- Could a guard clause flatten a nested `if`?
- Does a deeply-nested branch belong in its own named helper?
- Is there control-flow duplication (parallel branches with the same shape)?

Cognitive-complexity reductions usually buy more readability per change than McCabe-only reductions — they make the code humanly cheaper, not just structurally simpler.

### Single Responsibility (Naming + Cognitive together)

A function should describe **one** thing. Two operational checks:
- Can you describe what this function does without using "and" or "or"?
- If the name were removed and only the body remained, would you arrive at the same name from the body alone?

When the answer is no, the function is doing multiple things — typically a sign to extract a helper or split it in two. SRP is the lens that decides *whether* to split; it is not a separate metric. SRP violations usually surface as a vague name (resolvable by Naming Evaluation) or as high cognitive complexity (resolvable by extracting a helper).

### Common Smells and Typical Refactoring Moves

Vocabulary for seeing structure, not a checklist to apply per pass. Curated for the TypeScript/functional context of this kata work:

| Smell | Typical Move |
|---|---|
| Duplicated Code | Extract Function / Pull Up |
| Long Function | Extract Function, Decompose Conditional |
| Long Parameter List | Introduce Parameter Object |
| Data Clumps (same params travelling together) | Introduce Parameter Object / Extract Type |
| Primitive Obsession (raw `number`/`string` where a domain type would help) | Replace Primitive with Type |
| Switch on Type / Long Conditional Chain | Replace Conditional with Map Lookup or Polymorphism |
| Magic Number / Magic String | Replace with Named Constant |
| Comments that explain *what* (not *why*) | Extract Function with a revealing name |
| Feature Envy (function uses another module's data more than its own) | Move Function |
| Speculative Generality (unused parameters, dead branches, premature abstraction) | Inline / Remove |

No thresholds. These lenses are vocabulary for seeing and naming — not a quota for how many smells must be addressed per refactor pass. One concrete improvement that lowers cognitive load or APP mass beats a long list of cosmetic touches.
```
